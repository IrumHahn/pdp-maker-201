## 1. 목적

RE:BRANDB Client Portal은 웹사이트 제작 진행 상황을 클라이언트별로 실시간 공유하고, 내부에서는 전체 프로젝트를 통합 관리할 수 있는 에이전시 전용 대시보드다.

이번 MVP는 빈파트너스 프로젝트를 실제 운영 데이터로 바로 사용할 수 있게 설계한다.

핵심 목표는 아래 4가지다.

- 클라이언트가 현재 진행 상태와 본인 액션을 즉시 이해할 수 있게 한다.
- 내부가 일정, 승인, 자료 수급, 범위 변경, 커뮤니케이션을 한 화면에서 통제한다.
- 계약서 기준의 분쟁 포인트를 시스템에서 선제적으로 관리한다.
- 이후 다른 클라이언트 프로젝트에 템플릿처럼 재사용 가능하게 만든다.

## 2. 계약서 기준 운영 규칙

아래 규칙은 첨부 계약서 `빈파트너스_디자인용역_계약서_최종_0309.pdf` 기준이다.

- 계약명: 주식회사 빈파트너스 공식 웹사이트 제작 용역
- 계약 기간: 2026-03-09 ~ 2026-04-20
- 계약 금액: 3,900,000원, VAT 별도
- 대금 조건: 계약 체결 후 3영업일 이내 100% 선입금
- 클라이언트 자료 제공 마감: 계약 후 7영업일 이내
- 공식 협의 횟수: 총 3회
- 검수 통보 기한: 납품 후 3영업일 이내
- 무상 하자보수: 검수 완료 후 1개월
- 유지보수는 별도 계약
- 서드파티 월 사용료는 원칙적으로 클라이언트 부담
- AI 챗봇 운영비 3개월, Framer 호스팅 3개월은 수행사 부담 후 이후 클라이언트 전환

이 계약 조건 때문에 MVP에는 아래 기능이 반드시 들어가야 한다.

- 계약 정보 카드
- 자료 요청 및 제출 마감 관리
- 공식 회의 일정 및 기록
- 승인 요청과 승인 기한 카운트다운
- 검수 무응답 시 자동 승인 처리
- 하자보수 기간 카운트다운
- 유지보수 범위 외 요청을 별도 변경 요청으로 분리
- 서드파티 비용 전환 시점 관리

## 4. MVP 제품 방향

이번 MVP는 "프로젝트 운영 포털"에 집중한다. 실제 제작 사이트는 Framer에서 만들더라도, 클라이언트 포털은 별도 SaaS 형태로 운영한다.

빈파트너스는 이 포털의 첫 실제 사용 고객이 된다.

MVP에서 해결할 문제는 아래와 같다.

- 현재 진행 단계가 불명확하다.
- 클라이언트가 무엇을 언제까지 줘야 하는지 추적이 어렵다.
- 승인/수정 요청이 카톡, 메일, 전화로 흩어진다.
- 일정 지연 책임 소재가 불명확하다.
- 유지보수 범위와 추가 요청 범위가 섞인다.

## 5. 사용자 역할

### 5.1 내부 사용자

- `Owner`: 전체 고객, 전체 프로젝트, 권한, 시스템 설정 관리
- `Manager`: 담당 프로젝트 생성/수정/공유/승인 요청
- `Contributor`: 업데이트 작성, 파일 업로드, 메시지 응답

### 5.2 외부 사용자

- `Client Admin`: 자기 회사 프로젝트 전체 보기, 승인, 파일 업로드, 메시지 작성
- `Client Member`: 제한된 프로젝트 보기, 메시지 작성, 파일 확인

## 6. 핵심 기능 정의

### 6.1 인증/권한

- 클라이언트별 로그인 계정 발급
- 비밀번호 재설정 이메일
- 프로젝트 단위 접근 권한
- 내부/외부 표시 분리
- 마지막 로그인 기록

권장 방식은 `Supabase Auth 이메일/비밀번호 + 초대 링크`다.
아이디/비밀번호를 수기로 전달하는 구조보다 안전하고 운영이 쉽다.

### 6.2 클라이언트 대시보드

빈파트너스 로그인 후 첫 화면에 아래 카드가 보여야 한다.

- 현재 프로젝트 상태
- 전체 진척률
- 현재 진행 단계
- 다음 액션
- 승인 대기 항목
- 자료 요청 항목
- 최근 업데이트
- 최근 대화
- 다음 미팅 일정
- 주요 계약 정보

### 6.3 프로젝트 타임라인

기본 단계는 아래처럼 고정 템플릿으로 제공한다.

- 계약 완료
- 자료 수집
- 디자인 컨셉
- 기능 협의
- 디자인/개발 진행
- QA/수정
- 납품/검수
- 오픈
- 하자보수

각 단계에는 아래 속성이 필요하다.

- 시작일
- 종료일
- 상태
- 담당자
- 공개 여부
- 관련 파일
- 관련 승인
- 관련 메시지

### 6.4 작업 관리

프로젝트 내부 작업 단위를 `마일스톤`과 `태스크`로 나눈다.

- 마일스톤: 큰 단위의 고객 공유용 단계
- 태스크: 내부 작업 단위

상태값은 아래 5개면 충분하다.

- `not_started`
- `in_progress`
- `waiting_client`
- `blocked`
- `done`

### 6.5 공식 업데이트 로그

채팅과 별도로 "공식 진행 공유" 영역이 필요하다.

각 업데이트에는 아래 정보가 들어간다.

- 제목
- 요약
- 상세 내용
- 현재 단계
- 진행률
- 첨부 파일
- 클라이언트 액션 필요 여부
- 게시 시각

이 기능이 있어야 클라이언트는 "카톡 대화 전체"를 읽지 않아도 된다.

### 6.6 파일 관리

- 시안 파일 업로드
- 문서 업로드
- 링크형 자산 저장
- 버전 메모
- 클라이언트 가시성 설정

필수 분류는 아래 정도면 충분하다.

- `contract`
- `content`
- `design`
- `development`
- `delivery`
- `meeting`

### 6.7 승인 플로우

빈파트너스 계약에는 검수 기한이 명시되어 있으므로 승인 기능은 MVP 핵심이다.

승인 요청 유형:

- 디자인 컨셉 승인
- 기능 범위 승인
- 최종 납품/검수 승인
- 변경 요청 승인

승인 상태:

- `pending`
- `approved`
- `rejected`
- `approved_by_timeout`

규칙:

- 납품 후 3영업일 내 회신 없으면 `approved_by_timeout`
- 승인 시각과 승인자 저장
- 승인 전 버전과 승인 대상 파일 연결

### 6.8 메시지/대화

- 프로젝트별 메시지 스레드
- 파일 첨부
- 읽음 여부
- 관리자 내부 메모와 외부 메시지 분리
- 새 메시지 이메일 알림

실시간성이 중요하므로 MVP에서도 새 메시지와 상태 변경은 즉시 반영한다.

### 6.9 미팅 관리

계약서 기준 공식 협의 3회를 시스템에 고정 생성한다.

- 1차: 디자인 컨셉 공유
- 2차: 기능 협의
- 3차: 최종 완료 및 오픈 점검

미팅마다 아래를 기록한다.

- 일정
- 참석자
- 안건
- 결정사항
- 후속 액션

### 6.10 변경 요청 관리

계약서상 유지보수/추가 기능은 별도 계약 대상이므로 이 기능이 매우 중요하다.

클라이언트가 요청을 등록하면 아래 항목으로 관리한다.

- 요청 제목
- 요청 내용
- 범위 구분
- 계약 범위 포함 여부
- 일정 영향
- 비용 영향
- 내부 검토 메모
- 최종 승인 상태

범위 분류:

- `included`
- `maintenance`
- `new_scope`

### 6.11 하자보수/운영 전환

오픈 이후 바로 끝나지 않는다. 하자보수 1개월과 서드파티 비용 전환이 남아 있다.

따라서 프로젝트 종료 카드에 아래 항목을 포함한다.

- 하자보수 종료일
- Framer 결제 전환일
- AI 챗봇 비용 전환일
- 뉴스레터 플랫폼 운영 주체
- 예약 캘린더 운영 주체
- 교육 완료 여부

### 6.12 관리자 어드민

관리자 화면에는 아래 기능이 필요하다.

- 전체 프로젝트 목록
- 상태별 필터
- 지연 프로젝트 표시
- 승인 대기 목록
- 자료 미제출 고객 목록
- 프로젝트 생성/복제
- 클라이언트 계정 생성/비활성화
- 공통 마일스톤 템플릿 관리
- 활동 로그

## 11. 기술 아키텍처

### 11.1 배포 구성

- 프론트엔드: Netlify
- API 서버: Hostinger VPS
- DB/Auth/Storage/Realtime: Hostinger VPS 내 Supabase
- 이메일 발송: Resend

### 11.2 권장 구조

- `apps/web`: Next.js 기반 관리자/클라이언트 포털
- `apps/api`: NestJS 기반 API
- `Postgres`: Supabase Postgres
- `Auth`: Supabase Auth
- `Storage`: Supabase Storage
- `Realtime`: Supabase Realtime
- `Email`: Resend

### 11.3 왜 이 구성이 맞는가

- Netlify는 Next.js 프론트 배포와 프리뷰 링크 운영이 쉽다.
- Hostinger VPS는 API와 Supabase를 함께 운용할 수 있다.
- Supabase는 인증, DB, 파일, 실시간 기능을 MVP 속도에 맞게 제공한다.
- Resend는 승인 요청, 비밀번호 재설정, 새 메시지 알림 같은 트랜잭션 메일에 적합하다.

### 11.4 운영 주의사항

Supabase를 VPS에 셀프호스팅하면 관리 포인트가 늘어난다.
따라서 아래 운영 조건을 전제로 한다.

- 최소 4 vCPU / 8GB RAM 이상 권장
- 일일 DB 백업
- 스토리지 용량 모니터링
- SSL 설정
- 장애 대응용 관리자 계정 분리
- 운영 로그 보관

MVP 속도만 보면 managed Supabase가 더 간단하지만, 이번 계획서는 사용자가 지정한 `Hostinger VPS + Supabase` 기준으로 작성한다.

## 15. 성공 기준

빈파트너스 프로젝트에서 아래가 되면 MVP는 성공이다.

- 클라이언트가 카톡 없이도 현재 진행 상태를 이해할 수 있다.
- 자료 요청, 승인 요청, 검수 요청이 시스템 안에서 닫힌다.
- 내부가 지연 원인과 책임 구간을 명확히 추적할 수 있다.
- 유지보수 범위와 추가 요청 범위가 분리된다.
- 같은 구조로 다음 클라이언트를 10분 내 신규 생성할 수 있다.

## 16. 결론

이번 MVP는 단순한 "진척률 보드"가 아니라, 에이전시 프로젝트 운영 시스템이어야 한다.

특히 빈파트너스 계약서 기준으로 보면 아래 5가지를 반드시 제품화해야 한다.

- 자료 제출 마감 관리
- 공식 협의 3회 기록
- 검수 3영업일 승인 로직
- 유지보수 범위 외 요청 분리
- 서드파티 비용 전환 관리

이 기준으로 개발하면 빈파트너스에서 바로 실사용 가능하고, 이후 RE:BRANDB의 표준 프로젝트 포털로 확장할 수 있다.
